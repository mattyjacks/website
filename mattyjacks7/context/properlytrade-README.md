## Overview

ProperlyTrade is a modern, high-end SaaS demonstration website showcasing a fictional all-in-one
trading infrastructure platform for prop trading firms and brokerages. The site was designed and
developed as a portfolio piece to demonstrate advanced web development capabilities including:

- Premium dark-themed fintech UI/UX design with glassmorphism
- Complex multi-page Next.js 16 application architecture with App Router
- AI-powered chatbot integration with a Pi-based credit system
- 80+ site hardening improvements across security, accessibility, SEO, and performance
- Smooth Framer Motion animations and micro-interactions
- Fully responsive design across all breakpoints
- SEO-optimized page structure with proper metadata and Open Graph tags
- Third-party API integration (OpenAI, OpenRouter)
- Client-side state management for credits and API keys
- Legal compliance pages (Privacy Policy, Terms of Use)
- Custom error, 404, and loading pages
- Print stylesheet support
- DNS prefetch and preconnect hints for performance
- Comprehensive security headers

The fictional product depicted on this site is an "all-in-one infrastructure platform for trading
firms" that would provide CRM, risk management, trading APIs, and automation tools. **None of these
features are real** - they exist purely as UI demonstrations.

## AI Trading Chatbot

### Overview

The AI Trading Chatbot is an experimental feature that allows users to have conversations about
trading strategies, market analysis, chart patterns, risk management, and more. It is powered by
GPT-5.4 via either the OpenAI API or OpenRouter API.

> **CRITICAL DISCLAIMER**: The AI chatbot is experimental and untested. It does NOT provide
> financial advice. All responses are AI-generated and may be inaccurate. Never make trading
> decisions based solely on AI output. Use at your own risk.

### Architecture

```
User Browser                    Next.js Server                 AI Provider
+-----------+                   +---------------+              +----------+
|           |  POST /api/chat   |               |  API Call    |          |
| ChatWidget| ----------------> | route.ts      | -----------> | OpenAI   |
|           |                   | (validates    |              | or       |
|           | <---------------- |  input, adds  | <----------- | OpenRouter|
|           |  JSON response    |  system prompt)|  completion  |          |
+-----------+                   +---------------+              +----------+
      |
      v
  localStorage
  (credits, keys,
   chat history)
```

### Flow

1. User navigates to `/chat`
2. Disclaimer gate appears - user must accept risks before proceeding
3. User enters a message in the chat input (max 2000 chars, with live counter)
4. Client-side credit check occurs (deduct 1 credit or trigger overage warning)
5. Message is sent to `/api/chat` API route with conversation history
6. Server validates input (type, length, array structure) and prepends system prompt
7. Request is forwarded to OpenAI or OpenRouter API with GPT-5.4 model
8. Response is returned and displayed in the chat interface
9. Credit balance is updated in localStorage

### System Prompt

The AI operates under a carefully crafted system prompt that:

- Identifies it as "ProperlyTrade AI, an experimental trading assistant"
- Explicitly states it does NOT provide financial advice
- Reminds users about trading risks
- Allows discussion of technical analysis, fundamental analysis, risk management
- Frames specific levels as "areas of interest" not recommendations
- Supports Forex, Futures, Crypto, Equities, and Options

### Provider Support

#### OpenAI (Default)
- Uses the official `openai` npm package
- Model: `gpt-4o` (mapped from GPT-5.4 latest)
- Direct API access with user-provided or server-side API key
- Lower latency for users with direct OpenAI keys

#### OpenRouter
- Uses REST API calls to `https://openrouter.ai/api/v1/chat/completions`
- Provides access to multiple model providers through a single API
- Includes `HTTP-Referer` and `X-Title` headers for proper attribution
- Useful for users who prefer OpenRouter's pricing or model selection

### BYOK (Bring Your Own Key)

Users can provide their own API keys for either provider:

- Keys are stored in browser `localStorage` (never on the server)
- Keys are sent to the server API route only for the duration of the API call
- Using your own key grants **\u03c0x** daily credits (~31.4159.../day instead of 10/day)
- The BYOK multiplier is exactly Pi (3.141592653589793238462643383279502884197169)
- Credit displays show the integer (floor) value; hover over the number to see the full Pi-precision value
- Keys can be toggled on/off and cleared at any time
- The API key settings panel shows/hides keys with eye toggle buttons

### Chat Widget Features

- **Suggested prompts**: 4 pre-written trading questions to get started
- **Message history**: Full conversation displayed with user/assistant avatars
- **Loading state**: Animated spinner with aria-live region for screen readers
- **Overage warnings**: Alert when free credits are exhausted with cost info
- **Credit counter**: Real-time display of remaining credits with Pi hover tooltip
- **Character counter**: Live character count on input (max 2000)
- **Keyboard shortcuts**: Enter to send, Shift+Enter for newline
- **Error handling**: Graceful error display for API failures
- **Accessible send button**: aria-label for screen readers

## Credit System

### Overview

The credit system controls access to the AI chatbot. It is entirely client-side (stored in
`localStorage`) for this demo. In a production environment, this would be backed by a database.

### Credit Allocation

| Tier | Daily Free Credits | Max Accumulated | Overage Rate |
|------|-------------------|-----------------|-------------|
| Free | 10 | 100 | $0.05/prompt |
| BYOK (Own Key) | ~31 (\u03c0 x 10) | ~314 (\u03c0 x 100) | $0.03/prompt |
| Pro ($49/mo) | ~17/day (500/mo) | 1,000 | $0.02/prompt |

> **Easter egg**: The BYOK multiplier is exactly **\u03c0** (Pi). You see the integer value (31) in the UI,
> but hover over any BYOK credit number and you will see the full value:
> `31.41592653589793...` with a note that you get **\u03c0 x 10** credits per day with the
> self-provided BYOK API key setup bonus.

### How It Works

1. **Daily Refill**: At midnight UTC, users receive their daily credit allocation
2. **Accumulation**: Unused credits roll over up to the tier maximum
3. **10-Day Ramp**: Free tier reaches max 100 credits after 10 days of accumulation
4. **BYOK Bonus**: Providing your own API key multiplies your allocation by \u03c0 (Pi) instantly
5. **Overage**: When credits run out, users are warned but can continue at per-prompt cost
6. **No Lockout**: Users are never locked out - overages are always allowed

### Implementation Details

The credit system is implemented in `lib/credits.ts` with these key functions:

```typescript
loadCredits()                // Load state from localStorage, refill if new day
saveCredits()                // Persist state to localStorage
useCredit()                  // Deduct 1 credit, return allowed/overage status
setOwnApiKey()               // Toggle BYOK mode, adjust credit allocation
getOverageCost()             // Calculate cost for overage credits used
getByokDailyCreditsDisplay() // Get integer + full Pi display values + formula
getByokMaxCreditsDisplay()   // Get integer + full Pi display for max credits
```

### State Shape

```typescript
interface CreditState {
  balance: number;              // Current credit balance
  lastRefillDate: string;       // ISO date of last refill (YYYY-MM-DD)
  totalDaysAccumulated: number; // Days since first use
  hasOwnApiKey: boolean;        // Whether BYOK is active
  overageCreditsUsed: number;   // Overage credits used today
}
```

## Accessibility

### Standards Followed

The site implements WCAG 2.1 Level AA accessibility patterns including:

- **Keyboard Navigation**: All interactive elements are keyboard-accessible with visible focus indicators
- **Screen Reader Support**: Proper ARIA attributes throughout (`aria-label`, `aria-expanded`, `aria-controls`, `aria-current`, `aria-hidden`, `aria-live`, `role`)
- **Skip Navigation**: Skip-to-content link visible on keyboard focus
- **Semantic HTML**: `<main>`, `<nav>`, `<footer>`, `<section>` elements used correctly
- **Form Accessibility**: All form inputs have associated `<label>` elements via `htmlFor`/`id`
- **Reduced Motion**: `prefers-reduced-motion` media query disables animations
- **Focus Indicators**: Custom `focus-visible` ring styles on all interactive elements
- **Color Contrast**: Text/background ratios meet AA standards
- **Print Support**: Dedicated print stylesheet for readable printed output

| Component | Features |

## Animations

### Framer Motion Patterns

#### Fade-in-up (Most Common)
```tsx
initial={{ opacity: 0, y: 20 }}
animate={{ opacity: 1, y: 0 }}  // or whileInView
transition={{ duration: 0.5 }}
```

#### Staggered Children
```tsx
// Parent
variants={{ visible: { transition: { staggerChildren: 0.1 } } }}

// Children
variants={{
  hidden: { opacity: 0, y: 20 },
  visible: { opacity: 1, y: 0 }
}}
```

#### Slide-in (Chat Messages)
```tsx
initial={{ opacity: 0, y: 10 }}
animate={{ opacity: 1, y: 0 }}
transition={{ duration: 0.3 }}
```

#### Expand/Collapse (FAQ, API Settings)
```tsx
<AnimatePresence>
  {isOpen && (
    <motion.div
      initial={{ opacity: 0, height: 0 }}
      animate={{ opacity: 1, height: "auto" }}
      exit={{ opacity: 0, height: 0 }}
    />
  )}
</AnimatePresence>
```

- **Pulse**: Background gradient orbs (`animate-pulse`)

When `prefers-reduced-motion: reduce` is set:

## Security

### Security Headers

All responses include the following security headers configured in `next.config.ts`:

| Header | Value | Purpose |
|--------|-------|---------|
| `X-Content-Type-Options` | `nosniff` | Prevent MIME type sniffing |
| `X-Frame-Options` | `DENY` | Prevent clickjacking |
| `X-XSS-Protection` | `1; mode=block` | XSS filter |
| `Referrer-Policy` | `strict-origin-when-cross-origin` | Control referrer info |
| `Permissions-Policy` | `camera=(), microphone=(), geolocation=()` | Disable device APIs |
| `X-DNS-Prefetch-Control` | `on` | Enable DNS prefetching |

Additionally:
- `poweredByHeader: false` hides the `X-Powered-By: Next.js` header

### API Key Handling

- Server-side API keys are stored in environment variables (never exposed to client)
- User-provided API keys are stored only in browser localStorage
- Keys are transmitted to the server only during API calls (not persisted server-side)
- API route validates all input before processing

### Input Validation

The `/api/chat` API route validates:
- `messages` must be a non-empty array
- Each message must have `role` (string) and `content` (string) properties
- Individual messages capped at 10,000 characters
- Conversation history limited to 50 messages
- `provider` must be `"openai"` or `"openrouter"`

### Content Security

- AI system prompt constrains chatbot behavior and disclaimers
- All user-facing AI output includes disclaimer context
- Legal documents clearly marked as AI-generated
- Demo status prominently displayed across all pages

### Third-Party Security

- **Vercel**: SOC 2 compliant hosting with automatic HTTPS
- **Cloudflare**: DDoS protection, DNS security
- **Supabase**: Row-level security, encrypted connections
- **OpenAI/OpenRouter**: API keys scoped to specific capabilities

## Performance

### Optimization Strategies

1. **Font Loading**: Inter font loaded via `next/font/google` with `display: swap` (no FOIT)
2. **Component Splitting**: Each section is a separate component for code splitting
3. **Client/Server Split**: Only interactive components marked as `"use client"`
4. **Image Optimization**: No external images - all visual elements are CSS/HTML
5. **Animation Performance**: `whileInView` with `once: true` prevents re-animation
6. **Viewport Triggers**: Animations only fire when elements enter viewport
7. **Minimal Dependencies**: Only essential packages installed
8. **DNS Prefetch**: Pre-resolves `api.openai.com` and `openrouter.ai` domains
9. **Preconnect**: Early connection to Google Fonts
10. **Content Visibility**: `content-visibility: auto` on heavy sections for lazy rendering
11. **Turbopack**: Used in development for fast HMR
12. **Cache Components**: Next.js 16 feature for component-level caching
